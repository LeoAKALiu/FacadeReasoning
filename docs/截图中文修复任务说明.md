# 任务：修复说明书中英文截图的中文显示为方框

## 一、问题描述

本项目需要为《软件使用说明书》自动生成 38 张配图（图1.png～图38.png），并嵌入到 Word 文档 `软件使用说明书.docx` 中。

**当前现象**：在无中文字体的 Linux 服务器上，用 Puppeteer 截取的截图里，**所有中文都显示为方框（□□□）**，英文和数字正常。生成的 Word 中插图因此无法用于软著等正式材料。

**目标**：在本地电脑（或任意环境）上得到 **38 张中文显示正常的截图**，并重新生成带正确插图的 `软件使用说明书.docx`。

---

## 二、项目相关文件与流程

| 说明 | 路径/命令 |
|------|------------|
| 截图脚本（Node + Puppeteer） | `scripts/screenshot-manual.js` |
| 截图输出目录 | 项目根目录下的 `screenshots/`，文件名为 `图1.png`～`图38.png` |
| Word 生成脚本（Python） | `scripts/gen_manual_docx.py` |
| 说明书 Markdown 源稿 | `rz_polished.md` |
| 最终 Word 输出 | 项目根目录下的 `软件使用说明书.docx` |
| 插图与流程说明 | `插图清单.md` |

**标准流程**：

1. 确保应用已运行（本地 `npm run dev` 或访问已部署地址）。
2. 执行截图脚本，生成 `screenshots/图1.png`～`图38.png`。
3. 执行 Word 生成脚本，将上述截图嵌入并输出 `软件使用说明书.docx`。

## 三、已尝试过的方案（在服务器上仍无效）

1. **在截图脚本中注入网页中文字体**  
   在 `scripts/screenshot-manual.js` 中，在每次截图前：
   - 通过 jsDelivr CDN 加载 `@fontsource/noto-sans-sc`（400、700 字重）；
   - 注入样式强制全页使用 `font-family: "Noto Sans SC", ...`；
   - 使用 `document.fonts.ready` 并等待约 400ms 后再截图。  
   在目标 Linux 服务器上，截图中的中文仍为方框（可能因网络或字体未正确应用）。

2. **在服务器上安装系统级中文字体**  
   执行：`sudo yum install -y fontconfig wqy-zenhei-fonts && fc-cache -fv`。  
   完成后重跑截图脚本，中文仍为方框（可能 Chromium 未正确选用该字体）。

因此希望**在本地电脑**（如 Windows/macOS，通常自带或易安装中文字体）用 Codex / Claude Code 协助排查或改脚本，使截图中的中文正常显示。

## 四、本地执行前的准备

1. **环境**  
   - Node.js（建议 14+，当前项目用 Puppeteer 16.2.0，支持 Node 16）。  
   - Python 3，并安装：`pip install python-docx`（见 `scripts/requirements-docx.txt`）。  
   - 已安装依赖：`npm install`。

2. **应用需可访问**  
   - **方式 A**：在本地启动应用后对本地地址截图。  
     ```bash
     npm run dev
     ```  
     默认 `http://localhost:3000`，可改端口如 `npm run dev -- -p 3001`。  
   - **方式 B**：对已部署地址截图（不启动本地 dev），例如：  
     ```bash
     set BASE_URL=http://47.98.176.41:3001
     ```
     （Windows CMD；PowerShell 用 `$env:BASE_URL="http://47.98.176.41:3001"`；macOS/Linux 用 `export BASE_URL=http://47.98.176.41:3001`。）

3. **截图脚本命令**  
   ```bash
   npm run screenshots
   ```  
   或直接：  
   ```bash
   node scripts/screenshot-manual.js
   ```  
   使用已部署地址时需先设置 `BASE_URL`（见上）。  
   输出目录：项目根目录下的 `screenshots/`，文件名为 `图1.png`～`图38.png`。

4. **生成 Word**  
   ```bash
   npm run manual:docx
   ```  
   或：  
   ```bash
   python3 scripts/gen_manual_docx.py
   ```  
   会读取 `rz_polished.md` 和 `screenshots/图N.png`，生成 `软件使用说明书.docx`。

## 五、请协助尝试的方向

在本地电脑上，可依次尝试以下思路（由易到难），直到截图中的中文正常显示为止。

### 1. 直接在本机跑现有脚本（优先试）

本地 Windows/macOS 通常已有「微软雅黑」「PingFang SC」等中文字体，Chromium 可能直接使用系统字体，无需改代码。

- 启动应用（本地 dev 或使用 `BASE_URL` 指向部署地址）。  
- 执行：`npm run screenshots`（或按上节设置 `BASE_URL` 后执行）。  
- 打开 `screenshots/图1.png`、`图7.png` 等检查中文是否仍为方框。  
- 若已正常，直接执行 `npm run manual:docx` 生成最终 Word。

### 2. 使用有界面模式跑 Chromium（便于观察字体）

在 `scripts/screenshot-manual.js` 中，将 `puppeteer.launch` 的 `headless: true` 改为 `headless: false`，再运行截图脚本。  
这样会弹出浏览器窗口，可直观看到页面中文是否正常；若正常，再截的图也会正常。若本机有中文字体，此方式常能直接解决问题。

### 3. 延长字体加载与渲染等待时间

若怀疑是字体或页面未完全渲染就截图，可在 `ensureChineseFont(page)` 内或调用后增加等待，例如把 `setTimeout(..., 400)` 改为 800～1500 ms，或在 `document.fonts.ready` 后再多等 500～1000 ms，然后重跑截图。

### 4. 换用其他中文字体 CDN 或本地字体

当前脚本通过 jsDelivr 加载 Noto Sans SC。可尝试：

- 换用其他 CDN（如 Google Fonts 镜像、字节跳动 CDN 等）的 Noto Sans SC 或思源黑体；  
- 或将一份中文字体（如 Noto Sans SC 的 woff2）放到项目的 `public/fonts/`，在应用中通过 CSS 引入，并确保截图脚本打开的页面会加载该 CSS；脚本内再强制 `font-family` 使用该字体并等待 `document.fonts.ready`。

### 5. 为 Puppeteer 指定字体目录（若本机有字体）

在 `puppeteer.launch` 的 `args` 中增加 Chromium 字体路径（因系统而异），使 headless Chromium 使用本机已安装的中文字体。  
例如 Linux 下可尝试 `--font-render-hinting=none` 等；Windows/macOS 可查 Chromium 文档中关于 `--font-path` 或系统字体目录的说明（若有）。

### 6. 临时移除网页字体注入，仅依赖系统字体

若本机系统字体已能正确显示中文，可临时注释掉 `ensureChineseFont(page)` 的调用，仅用系统字体截图，排除 CDN 或注入逻辑带来的干扰。

## 六、验收标准

- `screenshots/` 下 38 张图（图1.png～图38.png）中的中文**无方框**，与浏览器中肉眼所见一致。  
- 执行 `python3 scripts/gen_manual_docx.py`（或 `npm run manual:docx`）后，生成的 `软件使用说明书.docx` 中对应插图里的中文显示正常，无成片方框。
